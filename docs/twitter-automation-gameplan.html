Inbox.GOP — Internal Strategy Document

Twitter/X Automation Gameplan

Version 1.0 Status: Pending Twitter API + Ad Approval April 2026
Overview

This document outlines the plan to build an automated Twitter/X posting system for Inbox.GOP. The goal is to drive daily organic traffic to public-facing pages by surfacing real political fundraising messages — emails and SMS — that candidates send to their supporters. Each post links back to a shareable, public URL on our platform, which will carry Google Ads once approved.

The system posts once per day, randomly selecting from messages sent within the last 24 hours that have not yet been posted. Only candidates are eligible (they have profile photos, summaries, and working directory links). No message is ever posted twice.

Why This Works

Ad Revenue Potential

Share page URLs are public and will carry Google Ads. A single viral tweet from a prominent candidate could send thousands of pageviews to an ad-enabled URL in hours.

Zero Marginal Cost

Content already exists in our database. The bot surfaces it automatically with no editorial effort. One post per day, fully automated once built.

Inherent Virality

Political fundraising messages are often outrage-inducing, funny, or alarming by design. They perform well on political Twitter without any additional editorializing.

SEO & Backlinks

Each tweet links to a canonical public URL on our domain. As posts get shared, we accumulate organic backlinks — which directly supports the Google Ads approval goal.

Content Selection Rules
1

Source: Emails and SMS Messages Only

Query both the Campaign (email) and SmsMessage tables. Entity-only posts are excluded for now — messaging content is more interesting and shareable.

2

Candidates Only

Filter to entities with type "candidate". These have profile photos, Ballotpedia summaries, party/state info, and working directory links — all of which make the share page rich and ad-friendly.

3

Sent Within the Last 24 Hours

Only messages where receivedAt (or createdAt) is within the past 24 hours are eligible. This keeps the content fresh and timely, which performs better on Twitter.

4

Never Repeat

A new boolean column — twitterPosted — is added to both tables. Once a message is posted, it is marked true and is permanently excluded from future selection.

5

Random Selection

Among all eligible messages, one is chosen at random. This ensures variety across candidates, parties, and states rather than always surfacing the same high-volume sender.

Share Token URL System

Each posted tweet links to a share token URL — a public, permanent link to a single email or SMS message. These URLs are already planned as part of the broader share feature. Example:

# Email share URL https://app.rip-tool.com/share/email/abc123def456 # SMS share URL https://app.rip-tool.com/share/sms/xyz789ghi012

The share token is generated on first post if one does not already exist. The page it links to should be public, SEO-optimized (SSR), and carry Google Ads units — making every tweet a potential ad impression funnel.

Note: The share page feature (generating token URLs + public share pages) needs to be built before the Twitter bot can go live. The bot code can be written in parallel; it just needs the share URL generation function to be available at runtime.
Tweet Caption Format

Captions are generated dynamically using the candidate's name, party, and message type. They are short, punchy, and include a direct link. Below are example formats:

👀 See what Tom Emmer (R-MN) just sent to supporters → https://app.rip-tool.com/share/email/abc123
🚨 Ted Cruz's campaign is texting voters. Here's exactly what they said → https://app.rip-tool.com/share/sms/xyz789
📬 Fresh from the Ron DeSantis fundraising inbox → https://app.rip-tool.com/share/email/def456
💰 This is what [Candidate Name] is sending out RIGHT NOW. Wild. → https://app.rip-tool.com/share/email/ghi012

Caption templates are stored as an array and selected randomly or rotated. Party (R/D) and state abbreviation are pulled from the entity record. Hashtags like #MAGA, #2026, or #PoliticalFundraising can optionally be appended to improve discoverability.

Technical Architecture
1

Vercel Cron Job

A scheduled route at /api/cron/twitter-post runs once daily at 10:00am EST via vercel.json cron config. Requires Vercel Pro plan (already on it).

2

Twitter API v2

Uses the POST /2/tweets endpoint with OAuth 1.0a credentials. The twitter-api-v2 npm package handles authentication. Requires a Twitter Developer App with Read+Write permissions.

3

Database Schema Change

Add twitterPosted: Boolean (default false) and twitterPostedAt: DateTime? to both the Campaign and SmsMessage Prisma models. One migration script required.

4

Environment Variables

Four new env vars added to Vercel: TWITTER_API_KEY, TWITTER_API_SECRET, TWITTER_ACCESS_TOKEN, TWITTER_ACCESS_TOKEN_SECRET. Stored securely, never in code.

5

Cron Security

The cron endpoint is protected by a CRON_SECRET header that Vercel injects automatically. Unauthenticated requests return 401.

// vercel.json — cron config { "crons": [ { "path": "/api/cron/twitter-post", "schedule": "0 15 * * *" // 10am EST = 15:00 UTC } ] }
Execution Flow (Daily)
1

Cron fires at 10:00am EST

Vercel calls /api/cron/twitter-post with the CRON_SECRET header.

2

Query eligible messages

Prisma queries Campaign and SmsMessage for records where: receivedAt is within last 24 hours, twitterPosted is false, and the associated entity is a candidate.

3

Random selection

One message is chosen at random from the combined result set. If no eligible messages exist, the cron exits gracefully with a 200 and logs a "nothing to post" message.

4

Generate or retrieve share token

If the message already has a shareToken, use it. If not, generate one (uuid), save it to the DB, and construct the public URL.

5

Build caption and post tweet

Select a caption template, inject entity name, party, and share URL. Post via Twitter API v2. On success, mark the message as twitterPosted: true and record twitterPostedAt.

6

Log result

Success and failure are both logged to the server console (visible in Vercel logs). No retries on failure — next day's cron will pick a different message.

Ad Revenue on Share Pages

Share pages (e.g. /share/email/abc123) are fully public, SSR-rendered, and not gated behind login. This makes them eligible for Google AdSense placements. Each tweet that performs well drives traffic directly to these pages, where ad impressions are recorded.

Scenario Est. Impressions/Day Notes
Quiet day, niche candidate 50–300 Still generates backlinks and crawl signals
Active news cycle, midsize candidate 500–3,000 Solid ad revenue at CPM of $2–5
Viral tweet, major candidate (Trump, DeSantis, Cruz) 10,000–100,000+ Significant revenue event; also great for AdSense approval signals
Important: Share pages must have Google Ads units placed on them before going live with the Twitter bot. The bot is the traffic driver — ads need to be ready to capture that traffic the moment it arrives.

Future Enhancements (Post-Launch)

Multiple Posts Per Day

Scale to 2–3 posts per day once the account has followers and engagement. Spread across morning, afternoon, and evening EST windows.

Entity Spotlight Posts

Once messaging posts are running well, add a weekly "entity spotlight" post: candidate photo, stats (X emails, Y SMS), and link to directory profile.

Trend Detection

If a candidate sends an unusually high volume of messages in a short period, trigger an out-of-schedule post: "Ted Cruz has sent 12 emails in 6 hours. See why."

Reply Engagement

Monitor reply sentiment on posted tweets. Popular posts can be quote-retweeted with follow-up context to extend the engagement cycle.
